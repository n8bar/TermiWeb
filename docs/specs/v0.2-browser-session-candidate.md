# 0.2 Browser Session Candidate

## Status

- Candidate only
- Not part of `0.1`
- May move later than `0.2` if dogfooding shows other priorities

## Intent

Add a second instance type alongside terminal instances:

- `Terminal`
- `Browser`

The goal is to let a remote device browse the web as if it were using the workstation, without remoting the entire desktop.

## First-Release Shape

The first browser-session release should stay narrow:

- one browser instance equals one tab
- workstation-hosted browser engine
- isolated per-instance browser profile
- address bar
- back
- forward
- reload
- maybe stop

The remote device should control and view that workstation-hosted browser session through TermiWeb's instance model.

## Non-Goals For The First Release

The first browser-session release should not try to ship:

- multiple tabs
- extension support
- personal browser profile integration
- Brave Sync integration
- reuse of an existing local browser window
- claims of a full desktop browser replacement

## Product Fit

This fits TermiWeb if it stays aligned with the current product shape:

- workstation-first
- instance-based
- remote access without full desktop remoting
- useful on a trusted private network first
- still compatible with more advanced operator-managed deployments later

## Open Questions

- Which browser engine should back the first release?
- How should browser rendering and input be transported to the client?
- How should downloads, uploads, clipboard, and auth-heavy sites behave?
- How should browser instances be isolated from one another?
