# 0.2 QR Connect Candidate

## 1. Status

1. `0.2` candidate.
2. Not part of `0.1`.

## 2. Intent

1. Add a quick connect affordance that helps a second device open the current TermiWeb endpoint without manual URL entry.
2. The feature should support the existing workstation-first model by making it easier to hand off from the workstation browser to another device on the same reachable address.

## 3. Core Direction

1. The affordance should live in the top bar.
2. It should expose a QR code for the current access address.
3. It should appear only when the current access address is not loopback.
4. Loopback-only views such as `127.0.0.1` or equivalent local-only access should not show the QR affordance.

## 4. Constraints

1. This is a convenience feature, not a replacement for understanding the current URL or LAN access path.
2. It should not appear when scanning it would obviously fail from another device because the current address is loopback-only.
3. It should not become a noisy permanent top-bar distraction when it is not useful.
4. It should stay compatible with the existing private-network-first product posture.

## 5. UX Direction

1. The control should be obvious but compact.
2. The QR code should be one tap or click away, not always expanded.
3. The feature should help a second device reach the same TermiWeb app, not necessarily the same instance selection unless the product later adds explicit deep-link behavior.
4. The UI should still show or preserve the human-readable address path alongside the QR-based shortcut.

## 6. Open Questions

1. Should the QR encode only the origin, or a deeper link into the current session context?
2. Should the QR affordance appear for hostname-based access, IP-based access, or both?
3. Should the feature include a copy-link companion action in the same UI?
4. How should the loopback check behave for hostnames like `localhost` versus resolved machine names?

## 7. Non-Goals

1. Any `0.1` commitment or release dependency.
2. Public-internet discovery or device pairing beyond the current address.
3. Replacing the normal visible host/IP cues in the UI.
4. Acting as a security feature rather than a connection convenience.
