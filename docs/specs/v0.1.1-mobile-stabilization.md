# 0.1.1 Mobile Stabilization Spec

## Scope

Version `0.1.1` is a patch release focused on the two must-have mobile fixes promoted into `M9`:

- Finding `13`: collapsed mobile instance controls are too small to use cleanly.
- Finding `14`: mobile keyboard text assistance can corrupt terminal input.

Hosted CI is intentionally out of scope for this patch release while TermiWeb remains a one-maintainer project. It is tracked as a future contributor-readiness candidate in [the project plan](../PLAN.md).

## Release Discipline

The `0.1.1` release candidate must pass the normal local verification commands:

1. `npm run typecheck`
2. `npm test`
3. `npm run build`
4. `npm run build:site`
5. `npm run package:release`

Manual verification must cover the task-specific mobile behaviors below before `0.1.1` ships.

## Finding 13: Collapsed Mobile Instance Controls

### Required Behavior

- In collapsed sidebar mode, the active instance must present one large touch target for selecting or expanding the active-instance controls.
- The collapsed active-instance row must not squeeze the columns control and close action into the always-visible collapsed rail.
- Width/columns controls and the close action should be available only after expanding or opening the active-instance controls from that larger touch target, and the opened controls must be physically larger than the collapsed rail instead of reusing cramped rail-sized buttons.
- The opened controls should align with the top of the active instance card without forcing focus into the column editor or opening the mobile keyboard.
- Tapping or clicking an inactive collapsed session should only select it; tapping or clicking the already active collapsed session should toggle its opened controls.
- The `Type` action must remain available from the collapsed sidebar.
- The close action must remain physically separate from the instance-selection hit target once expanded.
- The behavior must preserve browser-local sidebar state and must not introduce shared-session state for collapsed controls.

### Acceptance Checks

1. On a portrait phone viewport, the collapsed active-instance control is large enough to tap reliably.
2. The columns control is no longer cramped in the collapsed always-visible rail and opens as a large touch target.
3. The close action does not compete with the collapsed active-instance touch target.
4. Users can still select instances, reach columns controls from the already active collapsed instance, close the instance, and use `Type` without flashing or toggling the mobile keyboard.
5. Narrow mobile column presets preserve useful vertical context through explicit stable geometry pairs instead of shrinking rows only by the baseline width ratio.
6. Expanded desktop/sidebar behavior remains usable and is not regressed by the collapsed mobile change.

## Finding 14: Mobile Text Assistance

### Required Behavior

- Terminal input surfaces must request the strongest practical browser/OS suppression for text assistance.
- The mitigation should cover autocorrect, autocapitalize, spellcheck, predictive text, grammar assistance, smart punctuation, and suggestion overlays where browser hints allow it.
- Terminal input must not use password-style or credential-shaped fields as a text-assistance workaround because password-manager overlays are worse than the remaining best-effort keyboard assistance risk. This constraint applies to PTY/terminal capture only; the actual TermiWeb login remains normal password entry.
- If normal text capture still allows keyboard prediction to corrupt terminal input, the mobile-only fallback may use a non-credential input type that browsers treat as non-autocorrecting, such as `type="url"`, as long as shell entry remains usable.
- The hidden/live terminal input path and any fallback paste or type fields used for terminal entry should be reviewed.
- The fix must not break normal terminal typing, paste, modifier composition, or selection/copy behavior.

### Candidate Browser Hints

Use the best applicable subset for each input/editable surface:

- `autocapitalize="off"`
- `autocomplete="off"`
- `autocorrect="off"`
- `spellcheck="false"`
- `enterkeyhint` where useful for terminal entry
- `inputmode` only if it helps without restricting normal shell characters
- `aria` or role changes only when they improve semantics without making the terminal less usable
- password-manager and writing-assistant ignore hints when they are harmless on normal terminal text inputs

Avoid changes that block punctuation, symbols, whitespace, paste, or modifier-driven terminal input.

Do not use `type="password"` or another credential-shaped hidden input for live terminal capture. If the mobile fallback needs a different input type, prefer a non-credential type that disables autocorrect without inviting password-manager overlays. This does not change the app-auth login field, which should keep normal browser password semantics.

### Acceptance Checks

1. On a mobile browser, typing punctuation-heavy shell commands does not replay or duplicate previously typed text.
2. Predictive/autocorrect UI is materially reduced wherever the browser honors the available hints.
3. Normal command entry, paste, modifier keys, selection mode, and copy behavior still work.
4. The mitigation is documented honestly as best-effort browser/OS hinting rather than a guaranteed ability to disable every keyboard feature on every mobile keyboard.
5. Password-manager overlays do not attach to live terminal typing because live terminal capture does not present a password or credential field.

## Manual Mobile Verification

Run these checks before shipping `0.1.1`:

1. Portrait phone: collapsed sidebar active-instance controls are reachable and not cramped.
2. Portrait phone: expand/open active-instance controls, change columns, and close an instance without accidental taps.
3. Portrait phone: use `Type`, modifier keys, arrows, `Home`, `End`, paste, select, and copy.
4. Portrait phone: type punctuation-heavy commands such as `echo "a.b:c/path?x=1"` and confirm text is not replayed or duplicated.
5. Rotate phone between portrait and landscape and confirm the terminal layout remains usable.
6. Workstation browser: create, select, resize, and close instances to confirm desktop behavior did not regress.
