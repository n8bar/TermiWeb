# 0.1 Release Package Contract

## Purpose

Define the minimum packaging decisions that later `M6` work depends on, without freezing internal folder choices or launcher details too early.

## Release Form

- The primary `0.1` release artifact should be a Windows `x64` zip package.
- The package should be usable after unpacking into a user-writable location.
- The packaged release should not require a separately installed system Node runtime.
- A full MSI-style installer is not required for `0.1`.

## Required Package Contents

- The built TermiWeb client and server runtime needed to run the app.
- The production dependency set required by that shipped runtime.
- A root runtime `package.json` so the bundled Node process keeps the packaged server in its current ESM module mode.
- A bundled Node runtime suitable for the packaged server launch path.
- `Set Up TermiWeb.cmd` at the package root.
- `Enable TermiWeb Auto Start.cmd` and `Disable TermiWeb Auto Start.cmd` at the package root.
- `Start TermiWeb.cmd`, `Restart TermiWeb.cmd`, and `Stop TermiWeb.cmd` at the package root.
- `Uninstall TermiWeb.cmd` at the package root.
- A configuration template starting from `.env.example`.
- `1.Start-Here.md` at the package root.
- `LICENSE`, `DISCLAIMER.md`, and `THIRD_PARTY_NOTICES.md`.

## Excluded Package Contents

- `src/`
- `tests/`
- the internal engineering `docs/` tree
- dev-only dependencies
- local runtime state such as `.termiweb/`
- local scratch material such as `.cybercreek/`
- duplicated build output that is not part of the shipped runtime
- repo-only maintenance files that do not help a packaged user run TermiWeb

## Runtime Assumptions

- The unpacked package directory is the runtime home for the packaged release in `0.1`.
- `.env` should live beside `.env.example` in that package directory.
- Runtime state should continue living under `.termiweb/` relative to the package directory.
- The packaged release may keep the normal product defaults unless a later M6 item changes them deliberately.
- The packaged launch surface may request Windows elevation because `0.1` runs elevated shells only.
- Optional before-sign-in auto-start should preserve that elevated-only behavior instead of quietly starting a limited-privilege host.
- Side-by-side packaged and repo-checkout use should rely on the dev checkout moving off the default port rather than forcing the packaged release onto a nondefault port.


## Non-Goals

- Reproducing the full repo structure inside the release package
- Windows service packaging in `0.1`
