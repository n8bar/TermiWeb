# 0.1 Per-Instance Column Width

## Status

- Shipped in `0.1`
- Implemented as a shared per-instance width control in the active-instance header

## Intent

Allow a user to change the column width for a specific instance, while keeping that width shared across all attached devices for that instance.

This is an operator-controlled tradeoff feature. If someone changes the width and a TUI or shell session reflows badly, that consequence belongs to the shared instance and is not something the product should try to hide.

## Core Rules

- New instances still start at `80` columns.
- Column width is an instance property, not a device property.
- Changing width applies to the live instance for every attached device.
- Column width persists with instance metadata.
- The configured column count must always fit inside the visible `xterm` viewport width on each device.
- Font sizing is the mechanism that makes the configured columns fit inside that visible width.
- Rows remain the elastic dimension and should come from the terminal height available after width fit is satisfied.
- Outer layout or stage logic must not override the terminal-fit invariant by clipping columns or hiding part of the configured width.

## UI

- Add a topbar control that shows the current width, for example `Cols 80`.
- Place it near the active-instance header information rather than burying it in the sidebar or a global settings view.
- Clicking the control opens a compact popover for the active instance.
- The popover should offer common presets:
  - `80`
  - `100`
  - `120`
  - `160`
- The popover should also offer a custom numeric input and an explicit apply action.
- The popover should include a short note such as: `Applies to this instance for all attached devices.`

## Session Behavior

- Width changes should resize the live session immediately.
- Width changes should update the shared terminal state for all attached clients.
- If more than one client is attached, the UI should warn or confirm before applying the change.
- Width changes should survive instance restarts through persisted metadata, but they do not imply PTY resurrection.
- Width changes must not produce horizontal clipping or hidden columns inside the visible `xterm` viewport.
- Width changes must not regress the current phone/desktop behavior into page-wide overflow or other viewport-breaking layout failures.

## Shipped Notes

- The initial implementation clamps custom widths to a bounded sane range.
- The server stores width with instance metadata and rehydrates it on startup.
- Clients treat width as shared session state and only use the current instance width when fitting the terminal locally.
- The fit path uses xterm's own proposed column fit to size the font so the configured columns stay inside the visible xterm viewport width.

## Clarified Direction

- The fit model should stay simple:
  - the instance provides a configured column count
  - the terminal measures its visible `xterm` viewport width
  - font size adjusts so that all configured columns fit inside that width
  - rows then come from the height that remains available

## Non-Goals

- No per-device column width override
- No automatic "best fit" width based on viewport
- No attempt to preserve perfect layout for TUI programs after a width change
- No global setting that retroactively changes every existing instance

## Open Implementation Notes

- The exact allowed numeric range can stay implementation-defined, but it should be bounded to sane values.
- The initial implementation should favor a compact popover over a larger settings panel.
- The current `80`-column default remains the starting point for all newly created instances.
