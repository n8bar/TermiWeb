# 0.1 Persistent Auth Sessions

## Status

- Shipped in `0.1`
- Implemented as persisted server-side session records under `.termiweb/`

## Intent

Keep authenticated browser sessions created by the configured app password valid across TermiWeb server restarts so a page refresh or automatic recovery refresh does not force an unnecessary re-login.

## Core Rules

- The configured app password remains the primary access model in `0.1`.
- A successful login should survive a normal TermiWeb server restart until the session expires or the user logs out.
- Logout should still revoke the current session immediately.
- Session expiry should still be bounded by the configured TTL.
- The feature must preserve the current cookie-based browser flow.
- This does not introduce user accounts, per-user authorization, or Windows-integrated auth.

## Storage Model

- Persist server-side auth session records under `.termiweb/`.
- Store only the minimum metadata required to validate a session token after restart.
- Expired sessions should be cleaned up automatically.
- Restart-stable auth state should be treated as local runtime state, not committed repo content.

## Behavior

- If the server restarts and a browser still has a valid session cookie, `/api/auth/session` should continue to report `authenticated: true`.
- Automatic page recovery after restart should keep the user inside the app when the cookie is still valid.
- Invalid, expired, or revoked session tokens should still require a fresh login.
- Restart-stable auth should not claim PTY resurrection; it only preserves login state.

## Shipped Notes

- The initial implementation persists lightweight session token records to `.termiweb/auth-sessions.json`.
- The runtime keeps the existing app-password-backed cookie flow and restores valid tokens on server startup.
- Expired sessions are pruned automatically when the store is loaded, issued into, or validated.

## Non-Goals

- No Windows-integrated auth in `0.1`
- No collection of Windows credentials by the app
- No multi-user account system
- No browser-independent revocation model beyond the existing session token behavior
