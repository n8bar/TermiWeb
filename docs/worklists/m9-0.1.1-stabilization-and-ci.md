# M9 0.1.1 Stabilization And CI

Milestone Timing: Mirrors `M9` in [Project plan](../PLAN.md): start `2026-04-27`, target end `2026-05-10`.

This is the first `M9` worklist draft. It should be reviewed before execution is treated as approved direction.

## Desired Outcome

TermiWeb should leave `M9` with a credible `0.1.1` patch release: the must-have mobile fixes from Findings `13` and `14` shipped, a hosted CI baseline in place, and a clearer release discipline for follow-up stabilization work.

## 1. Work Breakdown

1. [ ] Define the `0.1.1` release discipline and CI baseline.
   1. [ ] Decide which checks must run in hosted CI for `0.1.1`.
   2. [ ] Decide when those checks run: pushes, pull requests, release branches, or a narrower rule if that is the right fit.
   3. [ ] Define the minimum pass bar for a `0.1.1` release candidate once hosted CI exists.

2. [ ] Implement the first hosted CI flow.
   1. [ ] Add the workflow or workflows needed to run the chosen hosted checks.
   2. [ ] Make sure the workflow output is readable enough to support normal debugging instead of being a ceremonial green badge.
   3. [ ] Confirm the repo can rely on that baseline going forward without contradicting the existing local-first verification guidance.

3. [ ] Make the new CI baseline prove itself on the current codebase before product fixes land.
   1. [ ] Run the hosted checks against the current `main` state and fix any baseline issues they expose.
   2. [ ] Document the resulting `0.1.1` verification expectations where they belong.

4. [ ] Fix Finding 13 for `0.1.1`.
   1. [ ] Define the collapsed mobile instance-control behavior that replaces the current too-small controls.
   2. [ ] Implement the chosen mobile layout and interaction changes.
   3. [ ] Verify the fix against the new CI baseline plus real mobile checks.

5. [ ] Fix Finding 14 for `0.1.1`.
   1. [ ] Identify the browser and input hints available for suppressing mobile text assistance in terminal entry.
   2. [ ] Implement the best supported mitigations without breaking normal terminal input behavior.
   3. [ ] Verify the fix against the new CI baseline plus real mobile checks.

6. [ ] Refresh the release surface and ship the `0.1.1` patch release.
   1. [ ] Update the release notes, package metadata, artifact references, and any versioned site or README references needed for `0.1.1`.
   2. [ ] Update any package-root docs, screenshots, or public copy that stop being honest once the `0.1.1` fixes land.
   3. [ ] Run the final local and hosted verification pass for the patch release.
   4. [ ] Publish `0.1.1` and close the milestone cleanly.

## 2. Verification Checklist

1. [ ] Hosted CI exists and is no longer just a future-candidate idea in the plan.
2. [ ] The hosted CI baseline is strong enough to catch obvious regressions without turning the project into process theater.
3. [ ] Finding `13` is fixed in a way that actually improves mobile usability instead of just reshuffling cramped controls.
4. [ ] Finding `14` is fixed in a way that materially reduces mobile text-assistance interference during terminal entry.
5. [ ] `0.1.1` ships as a real stabilization release and the public release surface stays aligned with what the patch release actually contains.
